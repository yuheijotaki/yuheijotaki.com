改善案取り入れるとさらに良くなる機能

1. focus-visible polyfill を削除する

場所 src/layouts/Layout.astro:50, package.json の focus-visible: ^5.2.1

:focus-visible は Baseline Widely Available（2022〜、全モダンブラウザで対応済）。 にもかかわらず import 'focus-visible/dist/focus-visible.min.js' を読み込み続けているため、無条件で 1.5 KB 程度の JS が全ページに配信されている。 :focus:not(.focus-visible) も _base.scss に残っており、両者を一緒に削除できる。

提案 ① Layout.astro の import 'focus-visible/...' を削除 ② _base.scss の :focus:not(.focus-visible) { outline: none; } を削除 ③ npm uninstall focus-visible

2. text-wrap: balance / pretty で日本語の改行品質を上げる

場所 src/components/Heading.astro, src/components/Article.astro の :global(h1)〜h6, :global(p)

見出しに text-wrap: balance、本文段落に text-wrap: pretty を当てると、日本語タイトルでありがちな「最終行に一文字だけ落ちる」現象や、段落末尾の孤児（runt）を抑制できる。 balance は Baseline 2024-05-13、pretty は Chrome/Safari 対応・Firefox 未対応だが progressive enhancement として無害。 

/* Heading.astro */
.heading { text-wrap: balance; }

/* Article.astro */
:global(:where(h1, h2, h3, h4, h5, h6)) { text-wrap: balance; }
:global(:where(p, blockquote, li)) { text-wrap: pretty; }

注意 全要素にグローバル適用しない（CPU コストが高い）。短い見出し・段落のみに限定。

3. content-visibility: auto で 511 記事ある blog 一覧を軽量化

場所 src/components/PostItem.astro（および src/pages/blog/index.astro 経由でレンダリングされる箇所）

/blog/ は draft でない記事を全件描画する作りで、現状 500 を超える <li> を SSG が出力している。各 PostItem は単純構造とはいえ DOM ノード数 × レイアウトコストはバカにならず、 初回ペイント後のスクロール時にスタイル再計算が走るリスクがある。

初期ビューポート外の PostItem に対し content-visibility: auto ＋ contain-intrinsic-size を当てることで、未描画分のレイアウト/ペイントを完全に skip できる。 

/* PostItem.astro */
.post-item {
  content-visibility: auto;
  contain-intrinsic-size: auto 4rem; /* 1 行分の想定高 */
}

注意 初期表示の above-the-fold（先頭数件）まで適用すると逆効果なので、 :nth-child(n + 11) 等でフォルトを切る方が安全。

4. color-scheme + light-dark() でテーマ実装を簡素化

場所 src/styles/foundation/_variables.scss, src/layouts/Layout.astro の inline script

現状の作りは :root[style*='--color-mode: dark'] を inline JS で attribute 操作し、その属性セレクタで CSS 変数群をオーバーライドする独自実装になっている。Baseline 2024 の light-dark() と color-scheme: light dark を組み合わせれば、CSS 側のオーバーライドブロックは不要になる。 

:root {
  color-scheme: light dark; /* 既に meta だけにある。CSS にも入れる */
  --color-primary: light-dark(#100f0f, #fffcf0);
  --color-secondary: light-dark(#fffcf0, #1c1a1a);
  /* ... 他のペアも同様に1行で定義 */
}

ユーザートグルの実装は、ガイドの推奨通り「<meta name="color-scheme"> の content を light / dark / light dark で書き換える」方式に揃えると、 style="--color-mode: dark" を直接 root に書く現方針より副作用が少ない。

参考 npx -y modern-web-guidance@latest retrieve "dark-mode"

5. Astro <ClientRouter /> で View Transitions を有効化

場所 src/layouts/Layout.astro, src/components/Main.astro（.transition-fade クラス）, src/layouts/Footer.astro（.transition-fade クラス）

すでに transition-fade という命名のクラスが Main / Footer に振られており、 フェード遷移を意図した痕跡がある。一方で Astro 6 の <ClientRouter />（Astro < 5 の <ViewTransitions /> 後継）は 現状未導入。ヘッダの <Head> 直後に置くだけで、サブセカンドのページ遷移と View Transitions API のフェードが有効になる。 

// Layout.astro
import { ClientRouter } from 'astro:transitions';

<head>
  <Head ... />
  <ClientRouter />
</head>

参考 npx -y modern-web-guidance@latest retrieve "cross-document-transitions"

6. Markdown 内 <img> へ自動属性付与の rehype プラグイン

場所 src/rehype-plugins/（新規プラグイン追加候補）

Article.astro 内の :global(img) は max-width: 100%; height: auto; までは設定するが、loading="lazy" / decoding="async" は付与されていない。 ブログ記事 511 本に手で属性追加するより、external-links.ts 同様の rehype プラグインで img 要素を visit して属性を付ける方が運用負荷が低い。 

// src/rehype-plugins/optimize-images.ts（新規）
import { visit } from 'unist-util-visit';

export default function optimizeImages() {
  return (tree) => {
    visit(tree, 'element', (node) => {
      if (node.tagName !== 'img') return;
      node.properties.loading ??= 'lazy';
      node.properties.decoding ??= 'async';
    });
  };
}

LCP 候補の hero イメージにだけ fetchpriority="high" を当てたい場合は、 記事先頭の最初の img を判定するロジックを追加する。

7. gsap / @unpic/astro は /demo/ 専用 — 本番依存からの分離検討

場所 package.json の dependencies, src/pages/demo/gsap.astro, src/pages/demo/unpic.astro

src/api/ 配下を grep した結果、gsap と @unpic/* は demo ページからのみ参照されており、 本番経路（トップ・ブログ・OG）から使われていない。/demo/* は本番ビルド成果物に含まれるかぎり GSAP の JS（〜80 KB gzip）は読み込まれる可能性がある。

提案 ① demo 配下を src/pages/demo/ から動的 import に切り替え、本番チャンクから分離 ② または /demo/ を本番ビルドから除外（dev だけで配信）するルーティング設定を入れる ③ それが難しい場合でも gsap を devDependencies に降格する選択肢を検討

8. OG 画像の Google Fonts 取得をビルド時に 1 回だけにする

場所 src/components/OgImage.tsx（getFontData()）

現在の getOgImage() は呼ばれるたびに fonts.googleapis.com/css2 を fetch して URL を正規表現抽出 → フォントファイルを再 fetch という構造。記事ごとにキャッシュミスすると Google Fonts への 往復が増える。画像キャッシュは効いているのでヒット時は問題ないが、初回ビルド時にぶれやすい。

提案 フォント ArrayBuffer をモジュールスコープにメモ化、または node_modules/.cache/astro-og/font.ttf として OG 画像キャッシュ同様にディスクキャッシュ。

9. テーマ初期化 inline script の簡素化

場所 src/layouts/Layout.astro:26-38, src/layouts/Footer.astro の ThemeManager

現状は inline script で localStorage を読み取り --color-mode という CSS 変数を root.style に書き、Footer の TypeScript クラスでトグルを管理。 ガイド推奨の方式（<meta name="color-scheme"> content を inline で書き換え＋ CSS 側で :root に color-scheme を設定）に揃えると、`window.colorMode` の global 副作用も取り払える。改善案 4 とセットで進めるのが筋。

10. 見出し ID のハッシュ短縮（衝突回避）

場所 src/rehype-plugins/add-heading-links.ts

SHA-256 の頭 8 文字（32 bit）を ID にしているため、生涯 511 記事のあいだに衝突確率は十分低いが、 鳩の巣的に 2 個目以降はぶつかる可能性ゼロではない。ガイドではないが 実装上のロバストネスとして 12 文字（48 bit）まで広げる、もしくは記事内で衝突検知してサフィックスを足す ロジックを足してもよい。優先度は低い。
