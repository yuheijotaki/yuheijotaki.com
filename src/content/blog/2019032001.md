<h2>3章 誤解されない名前</h2>

<blockquote><strong>名前が「他の意味と間違えられることはないだろうか？」と何度も自問自答する。</strong>

</blockquote>

<h3>3.1 例：<code>filter()</code></h3>

```
results = Database.all_objects.filter(&#34;year <= 2011&#34;)
```

<blockquote>この <code>results</code> には何が含まれているだろうか？

<ul>
<li><code>year &lt;= 2011</code> のオブジェクト</li>
<li><code>year &lt;= 2011</code> <strong>ではない</strong> オブジェクト</li>
</ul>

どちらかよくわからないのは、<code>filter</code> があいまいな言葉だからだ。

「選択する」のであれば、<code>select()</code> にしたほうがいい。「除外する」のであれば、<code>exclude()</code> にしたほうがいい。

</blockquote>

<h3>3.2 例：<code>Clip(text, length)</code></h3>

```js
const clip = (text, length) => {
↓↓↓
const truncate = (text, maxChars) => {
```

上記と同じ考え方で読み手に疑問を抱かせないようにする。  
最後から <code>length</code> 文字を削除するなら、<code>remove</code>  
最大 <code>length</code> 文字まで切り詰めるなら、<code>truncate</code>

<code>length</code> も

<ul>
<li>バイト数</li>
<li>文字数</li>
<li>単語数</li>
</ul>

なのかが明確でなく、名前に単位をつけたほうがいいので、<code>maxChars</code> にするとよい。

<h3>3.3 限界値を含めるときは <code>min</code> と <code>max</code> を使う</h3>

```js
const cartTooBigLimit = 10; //「未満」か「以下」かわかりづらい
↓↓↓
const maxItemsInCart  = 10;  // min や max を使うとわかりやすい
```

<h3>3.4 範囲を指定するときは <code>first</code> と <code>last</code> を使う</h3>

```js
const integerRangeStart  = 2;
const integerRangeEnd    = 4; // 終端を範囲に含めるのかが分かりづらい
↓↓↓
const integerRangeStart  = 2;
const integerRangeLast   = 4; // End ではなく Last を使う
```

<h3>3.6 ブール値の名前</h3>

```js
let readPassword        = true; // 「これから読み取るか」「すでに読み取っているか」分かりづらい
↓↓↓
let needPassword        = true; // or
let userIsAuthenticated = true; // こちらでもよさそう
```

<blockquote>ブール値の変数名は、頭に <code>is</code> <code>has</code> <code>can</code> <code>should</code> などをつけてわかりやすくすることが多い。

</blockquote>

例：<code>spaceLeft()</code> => <code>hasSpaceLeft()</code>

<blockquote>それから、名前を否定形にするのは避けたほうがいい。例えば、

</blockquote>

```js
disableSSL = false;
↓↓↓
useSSL     = true; // 肯定形にしたほうが声に出して読みやすい
```

<h3>3.7 ユーザの期待に合わせる</h3>

<code>get()</code> や <code>size()</code> には軽量なメソッドがユーザに期待されている。

<h2>4章 美しさ</h2>

<blockquote>優れたソースコードは「目に優しい」ものでなければいけない。

<ul>
<li>読み手が慣れているパターンと一貫性のあるレイアウトを使う。</li>
<li>似ているコードは似ているように見せる。</li>
<li>関連するコードをまとめてブロックにする。</li>
</ul>
</blockquote>

<h3>4.4 縦の線をまっすぐにする</h3>

<blockquote>列を「整列」させれば、コードが読みやすくなることがある。

</blockquote>

<strong>整列すべきなのか？</strong>

<blockquote>ぼくたちの経験では、プログラマが心配するほどの手間にはならない。もし手間になるようだったら、そのときは止めればいい。

</blockquote>

<h3>4.5 一貫性と意味のある並び</h3>

```js
const details  = '詳細情報';
const location = '場所';
const phone    = '電話番号';
const email    = 'Eメール';
const url      = 'URL';
```

<blockquote>であれば、ランダムに並べるのではなく、意味のある順番に並べるといい。例えば、こんな感じだ。

<ul>
<li>対応するHTMLフォームの <code>&lt;input&gt;</code> フィールドと同じ並び順にする。</li>
<li>「最重要」なものから重要度順に並べる。</li>
<li>アルファベット順に並べる。</li>
</ul>
</blockquote>

<h3>4.7 コードを「段落」に分割する</h3>

<blockquote>文章は複数の段落に分割されている。それは、

<ul>
<li>似ている考えをグループにまとめて、他の考えと分けるためだ。</li>
<li>視覚的な「踏み石」を提供できるからだ。これがなければ、ページのなかで自分の場所を見失ってしまう。</li>
<li>段落単位で移動できるようになるからだ。</li>
</ul>

これと同じ理由で、コードも「段落」に分けるべきだ。

</blockquote>

```js
const suggestNewFriends = (user, emailPassword) => {
  // ユーザーの友達のメールアドレスを取得する
  ...

  // ユーザーのメールアカウントからすべてのメールアドレスをインポートする。
  ...

  // まだ友達になっていないユーザを探す。
  ...

  // それをページに表示する
  ...
}
```

<h3>4.8 個人的な好みと一貫性</h3>

<blockquote>最終的には個人の好みになってしまうこともある。例えば、クラス定義の開き括弧の位置がそうだ。

</blockquote>

```
class Logger {
    ...
};
```

または、

```
class Logger
{
    ...
};
```

<blockquote>どちらを選んだとしても、コードの読みやすさに大きな影響はない。でも、この2つのスタイルを混ぜてしまうと、すごい読みにくいものになってしまう。

<strong>一貫性のあるスタイルは「正しい」スタイルよりも大切だ</strong>

</blockquote>
